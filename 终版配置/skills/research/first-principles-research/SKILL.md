# 第一性原理研究框架

## 核心原则

### 1. 动机优先（Motivation First）
**不要假设用户清楚自己想要什么**
- 目标/需求不清晰时，**必须先暂停讨论**
- 问清楚：时间范围？关注重点？用途？
- 示例："你要找的是今天/近 24 小时？股价相关/技术突破/全部类型？用于投资/研究？"

### 2. 路径最优（Optimal Path）
**发现非最短路径时直接指出**
- 如果当前方法不是最优，立即告诉用户
- 建议更好的办法（如：直接访问官方渠道 vs 多搜索引擎）
- 拒绝打补丁，追根溯源

### 3. 决策驱动（Decision-Driven）
**输出只保留影响决策的关键信息**
- 砍掉：背景铺垫、惯例解释、无关细节、重复信息
- 保留：核心事实、关键数据、影响逻辑、时间线
- 用符号标记重要性（🔴 核心、📊 数据、⚡ 逻辑）

### 4. 追根溯源（Root Cause）
**遇到问题先找根本原因**
- 搜索引擎被 block → 切换备用服务
- API 返回空 → 检查配置/权限
- 信息过时 → 交叉验证/过滤
- 每个决策都要能回答"为什么"

## 执行流程

### Step 1: 需求确认（必须）
在开始任何搜索/分析前，必须明确：
- **时间范围**：今天/近 24 小时/最近 3 天/本周
- **关注重点**：股价相关/技术突破/监管法律/全部类型
- **用途**：投资决策/研究/其他

**如果用户没说清楚，立即暂停并提问**

### Step 2: 多源搜索策略
当主渠道受阻时，按优先级尝试：

1. **首选 API**（Firecrawl）
   - 检测 blocked/captcha 关键词
   - 自动切换到备用 API
   - 使用独立 API Key

2. **多搜索引擎并行**
   - Google News → DuckDuckGo HTML → Bing
   - 直接访问官方渠道（ir.company.com, press releases）

3. **针对性查询**
   - 财经：`site:reuters.com OR site:bloomberg.com`
   - 科技：`site:electrek.co OR site:techcrunch.com`
   - 官方：`site:ir.company.com OR site:company.com/press`

### Step 3: 内容提取与验证
- 使用 scrape 获取详细内容
- 交叉验证多个来源
- 过滤过时/重复信息
- 提取关键数据点（价格、时间、数字）

### Step 4: 结构化输出
**必须包含**：
1. 核心新闻（按重要性排序，🔴 标记）
2. 关键数据（股价、预期、分析师评级）
3. 时间线（事件发生时间、影响周期）
4. 为什么关键（每则新闻的影响逻辑）

**禁止包含**：
- 背景铺垫
- 惯例解释
- 无关细节
- 重复信息

## 输出模板

```
## 🔴 核心新闻（今日最重要）

### 1. [标题]
- **时间**：[具体日期/时间]
- **内容**：[核心事实]
- **影响**：[对目标的影响逻辑]

### 2. [标题]
...

## 📊 其他重要信息
- [数据点 1]
- [数据点 2]

## ⚡ 为什么这些是关键？
1. [逻辑 1]
2. [逻辑 2]
```

## 常见陷阱

1. **假设已知需求**：用户没说清楚时，先暂停讨论
2. **过度依赖单一来源**：必须多源交叉验证
3. **信息过载**：只保留影响决策的关键信息
4. **时间模糊**：必须明确"今天"的具体日期
5. **路径非最优**：发现更好方法时直接指出

## 示例对话

**用户**："搜索特斯拉今天的关键新闻"

**错误做法**：直接开始搜索，返回一堆新闻

**正确做法**：
1. 先问："你要找的是：1) 今天（4 月 21 日）的新闻/报告？还是最近 24 小时？2) 关键的定义是什么？股价影响型/技术突破型/监管法律型/全部类型？3) 用途是什么？"
2. 用户确认："近 24 小时，全部类型"
3. 执行：Firecrawl → 多源搜索 → 结构化输出
4. 输出：只保留核心新闻 + 关键数据 + 影响逻辑

## 维护
- 定期更新 API 配置
- 记录被 block 的搜索引擎
- 优化查询关键词
- 根据用户反馈调整输出模板
