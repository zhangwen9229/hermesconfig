# Cron Job 拆分执行与文件上下文传递方案

## 适用场景
当需要执行多个有依赖关系的子任务，且需要：
1. 真正的串行执行（后序任务依赖前序任务结果）
2. 避免单次推送消息过长导致 WeChat 超时
3. 保持上下文完整性

## 核心方案：文件传递上下文

### 步骤 1：创建独立子任务
将大任务拆分为 N 个独立 cron 任务，每个任务间隔 5-10 分钟。

### 步骤 2：文件读写逻辑
- **任务 1-2（数据收集）**：查询结果写入临时文件
  ```
  文件路径：~/cron/output/{task_name}_context.md
  操作：write_file (创建或覆盖)
  ```
- **任务 3-N（分析处理）**：读取文件 + 追加分析
  ```
  文件路径：~/cron/output/{task_name}_context.md
  操作：read_file → 读取上下文 → patch_file (追加内容)
  ```

### 步骤 3：Prompt 设计
每个子任务的 prompt 必须包含：
```markdown
## 任务目标
[具体任务描述]

## 上下文读取
1. 读取文件：~/cron/output/{task_name}_context.md
2. 提取前序任务的关键信息
3. 基于上下文进行分析

## 输出要求
- 格式：纯 Markdown（无代码块嵌套）
- 字符限制：≤1000 字符
- 约束：无投资建议、无估值预测、无评级
```

### 步骤 4：执行时间配置
| 任务 | 执行时间 | 工作内容 |
|------|---------|---------|
| 任务 1 | T+0 | 数据收集 → 写入文件 |
| 任务 2 | T+5min | 数据收集 → 追加文件 |
| 任务 3 | T+10min | 分析 1 → 读取 + 追加 |
| 任务 4 | T+15min | 分析 2 → 读取 + 追加 |
| 任务 5 | T+20min | 分析 3 → 读取 + 追加 |
| 任务 6 | T+25min | 分析 4 + 总结 → 读取 + 追加 |

## 常见问题与解决方案

### Q1: 文件读取失败怎么办？
**A**: 在 prompt 中添加错误处理逻辑：
```markdown
如果文件不存在或读取失败：
1. 记录错误日志
2. 跳过当前任务或重试
3. 通知用户
```

### Q2: 推送失败怎么办？
**A**: 
- 每个子任务独立推送，避免单次消息过长
- 如果推送失败，将结果保存到文件，后续手动查看
- 优化内容结构，确保每个子任务≤1000 字符

### Q3: 如何确保上下文完整？
**A**:
- 使用 `patch_file` 而不是 `write_file` 追加内容
- 在文件头部添加时间戳和任务标识
- 每个任务读取完整文件，而不是只读取部分内容

## 示例：22 点全球市场前瞻分析

### 任务 1：国际热点抓取
```markdown
## 任务
搜索当日国际热点（A 股/港股/美股/外汇/大宗商品）

## 输出
- 创建文件：~/cron/output/market_analysis_context.md
- 写入内容：国际热点数据
```

### 任务 2：国内政策梳理
```markdown
## 任务
读取文件，搜索国内政策，追加内容

## 输出
- 读取：~/cron/output/market_analysis_context.md
- 追加：国内政策数据
```

### 任务 3-6：四视角分析
```markdown
## 任务
读取文件，基于完整数据进行分析

## 输出
- 读取：~/cron/output/market_analysis_context.md
- 追加：四视角分析 + 总结
```

## 验证步骤
1. 手动执行任务 1，检查文件创建
2. 手动执行任务 2，检查文件追加
3. 手动执行任务 3-6，检查上下文传递
4. 检查最终文件内容完整性
5. 验证 WeChat 推送（或文件保存）

## 注意事项
- 所有时间以北京时间（UTC+8）为准
- 避免在 prompt 中使用代码块包裹 Markdown
- 每个子任务独立推送，避免超时
- 文件路径需确保有写权限
- 定期清理临时文件（如：每天凌晨 2 点）
